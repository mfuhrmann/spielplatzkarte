# Proposal: Impressum and Privacy Pages

**GitHub issue:** #411  
**Status:** Draft

## Problem

spieli deployments have no Impressum or Datenschutzerklärung. Both are legally required in Germany for publicly accessible web services. In federated hub deployments, users have no way to identify the legal entity responsible for each regional backend.

## Goals

- Every spieli installation can expose an Impressum and Datenschutzerklärung.
- Maintainers who already host legal pages elsewhere can link out; those who don't get a working default generated from env vars.
- In hub mode, per-backend legal attribution is surfaced in the UI.
- All install types (`standalone`, `data-node-ui`, `data-node`) are covered with the same env vars and the same template logic.

## Non-goals

- i18n of legal page content (German only; i18n tracked in #157).
- Installer wizard prompts (follow-up).
- Rendering operator's hosting-provider log policies (out of spieli's knowledge).

## Design

### Two-layer content delivery

```
IMPRESSUM_URL / PRIVACY_URL set?
  yes → link out to external page (operator's own site)
  no  → serve generated default (from env vars + template)
```

Both layers use the same env vars:

| Env var | Required | Description |
|---|---|---|
| `IMPRESSUM_NAME` | yes | Responsible person or org |
| `IMPRESSUM_ORG` | no | Organisation name (if different from name) |
| `IMPRESSUM_ADDRESS` | yes | Street + city |
| `IMPRESSUM_EMAIL` | yes | Contact email |
| `IMPRESSUM_PHONE` | no | Contact phone |
| `IMPRESSUM_URL` | no | Override: link to external Impressum page |
| `PRIVACY_URL` | no | Override: link to external Datenschutz page |

### Impressum content

Generated by `docker-entrypoint.sh` from the structured env vars above. Straightforward HTML — name, address, contact.

### Datenschutz content

Pre-written template covering what spieli actually does:

- No user accounts, no tracking, no analytics cookies.
- OpenStreetMap data (public, attribution).
- Nominatim API (search bar — third-party or self-hosted).
- Panoramax (photo viewer — third-party).
- Browser geolocation (opt-in, never stored server-side).
- Contact details injected from same env vars.
- Explicit note: hosting infrastructure access logs are outside spieli's scope; operators must document their provider's log retention separately.

### Per-mode delivery

**`standalone` / `data-node-ui`**

`docker-entrypoint.sh` generates `impressum.html` + `datenschutz.html` into the nginx webroot. nginx serves them at `/impressum` and `/datenschutz`. `get_meta()` returns `impressum_url` and `privacy_url` as absolute URLs pointing to these paths.

**`data-node` (no web UI)**

Same env vars. Same template generation. `docker-entrypoint.sh` writes generated content into a `legal_content` DB table (type + content columns). A new PostgREST function `get_legal(type)` exposes the content. The hub fetches it on demand and renders it in a sanitized modal. `get_meta()` returns `impressum_url` / `privacy_url` as null when no override URL is set, signalling to the hub to fall back to `get_legal()`.

### UI

**ⓘ info button (standalone + hub — identical component)**

Small button on the map. Opens a modal containing:
- Impressum link / content
- Datenschutz link / content
- (Hub mode) hub's own legal attribution

Button and links are hidden if neither an override URL nor generated content is available (graceful degradation).

**InstancePanelDrawer (hub federation)**

Each backend row gets small § and 🔒 icons if the backend reports `impressum_url` / `privacy_url` (from `get_meta()`). Clicking opens the external page (new tab) or fetches `get_legal()` content and renders it in a sanitized modal. Icons are hidden if not configured.

### XSS boundary

Fetched HTML from backend `get_legal()` is rendered inside a sandboxed `<iframe srcdoc>` or passed through a strict HTML sanitizer before being placed in the hub DOM.

## API changes

### `get_meta()` — two new fields

```json
{
  "impressum_url": "https://example.com/impressum" | null,
  "privacy_url":   "https://example.com/datenschutz" | null
}
```

Null = no external override; hub falls back to `get_legal()`.

### New: `get_legal(type text)` → text

Returns the generated HTML for `type = 'impressum'` or `type = 'datenschutz'`. Used by the hub when `impressum_url` / `privacy_url` is null on a data-node backend.

## Open questions

- Sanitizer library choice for hub-rendered foreign HTML (DOMPurify is the standard pick for browser-side).
- Whether `get_legal()` should return `text/html` directly (requires PostgREST `Accept` header handling) or JSON `{ "content": "<html>..." }`.
