### Review Findings — Pass 1 (bmad-code-review, 2026-04-26)

Three review layers ran over the diff (Blind Hunter, Edge Case Hunter, Acceptance Auditor). 12 patches + 2 deferred + 1 dismissed.

#### Critical
- [x] [Review][Patch] Federation merge claim is factually wrong — `app/src/hub/hubOrchestrator.js:bucketToSuperclusterPoint` ships `[b.lon, b.lat]` into Supercluster, which kd-trees by **coordinates**, not by grouping cell. `architecture.md` ("merging keys on the grouping cell, not on `lon`/`lat`"), `design.md` D3 ("aligning on the shared cell grid"), and tasks §4.4's "Logical: ... keys on grid alignment" are all factually wrong. Rewrite to describe the actual mechanism — Supercluster on `lon`/`lat`, with the known interim consequence (jitter on backend toggle for shared cells, already noted in D3 paragraph 2 — but the framing as "merge contract preserved" misleads). [`docs/reference/architecture.md:74`, `openspec/changes/position-clusters-at-member-centroid/design.md:85-99`, `openspec/changes/position-clusters-at-member-centroid/tasks.md:25`]

#### High
- [x] [Review][Patch] JS bucketing uses `Math.floor(x / cellSize)` while PostGIS `ST_SnapToGrid(geom, size)` rounds to nearest — for the Fulda seed at z=7 they agree by coincidence (4 members fit one cell either way), but any future seed/bbox with a centroid in the upper half of a cell will diverge and break the `matching.length === 1` filter. Switch to `Math.round(x / cellSize) * cellSize`. [`tests/cluster-position.spec.js:541-552`]
- [x] [Review][Patch] Spec wording for "Clusters RPC honours bbox filter" was rewritten to "buckets whose grouping cell intersects the bbox are returned", but the SQL still filters by `ST_Intersects(ps.centroid_3857, b.geom)` (member centroid intersection). A cell straddling the bbox edge with one member-centroid inside and others outside will return a bucket with the inside members only — neither "cell intersects bbox" nor the legacy "centres fall within bbox" describes this. Reword to describe the actual member-centroid filter. [`openspec/changes/position-clusters-at-member-centroid/specs/tiered-playground-delivery/spec.md:30-34`]
- [x] [Review][Patch] `matching.length === 1` is brittle when two cells share the same `count` (e.g. two 2-member buckets at higher zoom). The SQL test sidesteps this by tie-breaking on distance to the reported `(lon, lat)`. Mirror that tie-break in the JS test. [`tests/cluster-position.spec.js:519-524`]
- [x] [Review][Patch] `ST_Centroid(ST_Collect(centroid_3857))` is not order-stable across plan changes (parallel scan, etc.). The new spec scenario "Clusters RPC bucketing is deterministic for a fixed dataset" demands `lon`/`lat` identity between calls — promote that from "deterministic in practice" to a guaranteed contract by adding `ORDER BY ps.osm_id` (or equivalent) inside the `ST_Collect`. [`importer/api.sql:278`]
- [x] [Review][Patch] Convex-hull / segment containment runs in WGS84 with `eps = 1e-9 deg` (~0.1 mm). For an N=2 (or collinear N≥3) bucket, the SQL function computes `ST_Centroid` in EPSG:3857 (linear metric mean) and reprojects to 4326; the reprojected midpoint is **not** on the WGS84 lon/lat segment between the two members (Mercator's `log(tan)` non-linearity offsets it perpendicular to the segment). The first 2-member bucket the test sees will fail spuriously. Run containment in 3857 metres with an mm-scale eps, or compute hulls/segment math after projection. Also relax the `polygon.length === 2` special-case to `polygon.length < 3` so collinear-N≥3 hulls (members along a road) are handled the same way. [`tests/cluster-position.spec.js:562-616`]

#### Medium
- [x] [Review][Patch] SQL regression's `≤ 1e-6 deg` tolerance comment is misleading — both sides of the comparison are produced by identical `ST_Transform` calls and should be bit-equal. Tighten to `1e-12` (numerical equality with safety margin) and rewrite the comment to say "bit-equality of two `ST_Transform` reprojections", not "physical-distance tolerance". [`dev/sql-tests/cluster-position.sql:18,110-114`]
- [x] [Review][Patch] `beforeAll` uses `test.skip` for both unreachable (network error) and reachable-but-broken (HTTP non-OK) backends, so a partial outage (db down, schema not loaded → 5xx on `get_meta`) silently skips the suite on every CI run. Distinguish: catch only network errors → skip; HTTP non-OK → fail. [`tests/cluster-position.spec.js:473-483`]
- [x] [Review][Patch] Spec scenario "for a bucket with a single member, the returned `lon` / `lat` equal that member's centroid reprojected to WGS84" is not exercised by either regression test — both filter `count >= 2`. Mathematically `ST_Centroid(ST_Collect(p)) == p` for n=1 holds today, but a future "weighted mean" refactor that normalised by `count + 1` (or similar) would silently break n=1 without test failure. Add an explicit single-member assertion. [`dev/sql-tests/cluster-position.sql:51-55`, `tests/cluster-position.spec.js:50-54`]
- [x] [Review][Patch] Spec scenario "Clusters RPC scales cell size with zoom" carries the invariant "total `count` across all buckets is identical across zoom levels". The whole `MODIFIED Requirement` is being re-asserted by this delta, so this scenario is in the contract too — but neither regression test calls the RPC at two zooms and compares the summed counts. Add a zoom-invariant assertion (e.g. `sum(count) at z=6 == sum(count) at z=10` over the same bbox). [`dev/sql-tests/cluster-position.sql`, `tests/cluster-position.spec.js`]
- [x] [Review][Patch] `pointInOrOnPolygon` segment-degenerate branch divides effectively by `lenSq`. For coincident endpoints (`a == b`, possible if a future seed has two playgrounds sharing a node), `lenSq === 0` and `dot <= lenSq + eps` is `0 <= eps` → true for **any** `p` on the line through that single point. Guard with `if (lenSq <= eps) return distSq(p, a) <= eps;`. [`tests/cluster-position.spec.js:610-616`]
- [x] [Review][Patch] SQL "≠ cell anchor" assertion can false-fail on a degenerate but plausible symmetric seed (members placed symmetrically about the snap target → centroid equals anchor). Replace the numerical inequality with a structural assertion ("the function's projection source is `bucket_centroid_3857`, not `cell`" via `pg_get_functiondef` regex), or document the seed assumption + improve the error message so a future failure surfaces "symmetry collision" not "pre-change regression". [`dev/sql-tests/cluster-position.sql:119-123`]
